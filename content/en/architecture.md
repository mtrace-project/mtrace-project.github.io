This page provides an overview of the architecture of **Mtrace** and its internal functioning.

## Container diagram
![Container diagram of Mtrace](../../images/containerDiagram.svg)

**Mtrace** provides two main commands: `mtrace create` and `mtrace run`, which allow you to create and run tests, respectively.

`mtrace create` generates a new test file with the `.mt.yaml` extension in the specified directory, providing a basic template that developers can adapt to their needs. While it is a supporting command and not strictly required to create a test file, it is highly recommended to prevent syntax errors and speed up the creation process.

`mtrace run` is the command that executes the tests passed as arguments or, if no arguments are provided, all tests located in the current working directory.
This command is the core of **Mtrace**, as it runs the tests and verifies that the traces generated by the system under test comply with the expected ones defined in the test files. Furthermore, it is the only command that interacts with external entities by interfacing with the following systems:
- **Observability backend**: collects and stores the traces generated by the system. **Mtrace** interfaces with it to retrieve the traces associated with the test execution and compares them against the expected traces.
- **System under test**: the application being tested. **Mtrace** interacts with it through an initial request called a *trigger*, aiming to generate a trace within the system.

## Component diagram
In this section, you can view a detailed diagram of the `mtrace run` command, including its internal components and their interactions.

![Component diagram of Mtrace](../../images/componentDiagram.svg)

### Run Test Controller
The component responsible for receiving the command via the CLI, interpreting, and validating the passed parameters and the current configuration.
Additionally, it orchestrates the parsing of the test files and their execution by interfacing with the `Test Parser` and the `Test Runner`.

### Parser
Responsible for reading and interpreting the test files, converting their structure from **YAML** format into an internal data structure used by the `Run Test Controller` for execution.

### Validator
Verifies that the internal data structure and the fields generated by the `Parser` are valid and meet expectations.
It also fills in any missing fields with **default values** (where possible) and reports validation errors to the `Run Test Controller`.

### Trace Test Runner
The orchestrator of individual tests. It manages all the necessary phases for running a test, interfacing with the `Setup Command Executor`, `Trigger Invoker`, `Trace Fetcher`, `Trace Comparator`, `Asserter`, and `Cleanup Command Executor`. It is also responsible for returning the test result to the `Run Test Controller`.

### Setup Command Executor
Executes the setup commands required prior to the test execution.
Setup commands are optional, can be defined inside test files, and are intended to prepare the environment before the *trigger* invocation (e.g., starting a Docker container required for the test).
For more details on setup commands and their configuration, please refer to the [Setup Commands]({{< ref "setupCommands" >}}) page.

### Trigger Invoker
Executes the *trigger* defined in the test file. Its goal is to generate a **traceId** and inject it into the system under test via a service call. Consequently, the **traceId** propagates through the system, making it possible to retrieve the trace later via the observability backend.
The *trigger* can be of different types, depending on the developer's needs and the system being tested.
The trigger types supported by **Mtrace** are listed on the [Triggers]({{< ref "../concepts/trigger.md" >}}) page.

### Trace Fetcher
Interfaces with the **observability backend** to retrieve the trace generated by the system following the *trigger* execution.
**Mtrace** supports various observability backends to accommodate different systems and developer preferences.
The supported backends are listed on the [Observability Backend Configuration]({{< ref "configure.md#observability-backend-configuration" >}}) page.

### Trace Comparator
Compares the characteristics of the retrieved trace with the expected trace (or sub-trace) defined in the test file. It provides clear and descriptive feedback regarding any discrepancies found.

### Asserter
Evaluates the assertions and queries defined in the test file. Assertions are boolean expressions applied to the collected trace; for the test to pass, all assertions must evaluate to true.
They are designed to provide greater flexibility and customization compared to simply defining an expected trace.
The assertion types supported by **Mtrace** can be found on the [Assertions]({{< ref "../concepts/assertion.md" >}}) page.

### Cleanup Command Executor
Executes the necessary cleanup commands after the test completes. This type of command is optional and aims to clean the environment after the execution of the *setup commands*.
Each type of *setup command* has a corresponding *cleanup command*, which can either be mandatory for the user to define or already set by default, depending on the setup type.
This component is crucial to allow repeated test executions without risking compromising or polluting the system under test.
